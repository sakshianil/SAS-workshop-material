# CLAUDE.md — Repository Reproducibility Contract

This file provides operational guidance for AI coding assistants and human
contributors working in this repository. Preserve the educational purpose,
data-provenance boundaries, reproducibility, and audit trail described below.

## Repository purpose

This repository contains two SAS OnDemand for Academics learning tracks:

1. `clinical-sas-workshop/` — a beginner-oriented, end-to-end clinical
   programming simulation.
2. `cdisc-pilot-mini-workshop/` — a compact learning companion based on selected
   public CDISC pilot artifacts.

The repository demonstrates programming workflows. It is not a validated
clinical system, sponsor deliverable, medical product, or regulatory submission.

## Data classification and confidentiality

### Synthetic workshop

All records under `clinical-sas-workshop/data/raw/` are artificially generated,
hypothetical, simulated data created for education.

- They do not describe real trial participants.
- Subject identifiers are fictional.
- Dates, adverse events, treatments, measurements, and outcomes are simulated.
- No PHI, PII, confidential sponsor data, or real clinical-study data should be
  added to this workshop.
- Describe these files as **synthetic data that simulate realistic clinical
  programming scenarios**, never as real patient data.

### CDISC pilot workshop

Files under these paths are selected public test/pilot artifacts obtained from
`cdisc-org/sdtm-adam-pilot-project`:

```text
cdisc-pilot-mini-workshop/source/cdisc/
cdisc-pilot-mini-workshop/reference_docs/
cdisc-pilot-mini-workshop/reference_programs/
```

- They are public reference material attributed to CDISC.
- They are not confidential sponsor data supplied by this repository owner.
- They must not be described as data generated by this project.
- Do not modify these source files. Their hashes are recorded in
  `cdisc-pilot-mini-workshop/SOURCE_MANIFEST.sha256`.
- Follow `cdisc-pilot-mini-workshop/ATTRIBUTION_AND_TERMS.md` and `LICENSES.md`.

### Prohibited content

Never commit:

- real patient or participant data;
- protected health information or personally identifiable information;
- confidential sponsor, CRO, site, or vendor material;
- credentials, tokens, account identifiers, or private server paths;
- proprietary protocol, SAP, CRF, standards text, or training transcripts.

## Reproducible local validation

From the repository root:

```bash
python3 tests/validate_repository.py
```

Expected high-level result:

```text
PASS: 11 modules, 51 SAS programs, and all source-data scenarios validated.
PASS: 5 labs, 29 SAS programs, and all selected CDISC source hashes/metadata validated.
PASS: repository structure, documentation, and both workshops validated.
```

This checks publication structure, local Markdown links, synthetic-data
scenarios, CDISC source integrity, lineage rules, and required learning files.

GitHub Actions runs the same command through
`.github/workflows/validate.yml`.

## Reproducing the synthetic workshop in SAS OnDemand

Prerequisite: access to SAS OnDemand for Academics and SAS Studio.

1. Upload the complete `clinical-sas-workshop/` directory.
2. Open `clinical-sas-workshop/setup/00_create_folders.sas`.
3. Set `project_root` to the exact uploaded server path.
4. Run:

```text
setup/00_create_folders.sas
setup/01_import_raw_data.sas
setup/02_verify_setup.sas
```

5. Confirm the SAS log contains:

```text
WORKSHOP SETUP CHECK: PASS
```

6. Complete modules `00` through `10` in numeric order.
7. For a final smoke test, run:

```text
setup/03_run_all_solutions.sas
```

Reference source counts:

- Demographics: 60 records
- Adverse events: 74 records
- Exposure: 223 records
- Vital signs: 492 records

Detailed instructions are in `clinical-sas-workshop/START_HERE.md`.

## Runtime evidence and outputs

User-generated proof that the programs run belongs under the root `outputs/`
directory—not inside the source workshop folders.

Use:

```text
outputs/
├── clinical-sas-workshop/
└── cdisc-pilot-mini-workshop/
```

For every evidence submission:

1. Copy and complete `outputs/OUTPUT_MANIFEST_TEMPLATE.md`.
2. Record the SAS environment and execution date.
3. Identify the exact Git commit used.
4. List the programs executed in order.
5. Include relevant logs and generated reports.
6. Record expected versus observed row counts.
7. State whether errors, warnings, conversion messages, or differences occurred.
8. Redact usernames, home-directory paths, email addresses, tokens, and any
   other identifying information before committing screenshots or logs.

Evidence may include:

- redacted `.log` or `.txt` files;
- HTML, RTF, or PDF reports;
- PNG screenshots of SAS Studio results;
- `PROC COMPARE` output;
- a completed run manifest;
- a checksum manifest for the submitted evidence.

Do not label a workflow as “verified in SAS” until runtime evidence exists for
the relevant commit.

## Change-control rules

- Preserve lesson structure: `README`, lesson, exercise, check, and solution.
- Keep exercise prerequisites aligned with concepts already introduced.
- Do not silently change expected record counts.
- Update validators and expected-output documentation when behavior changes.
- Do not weaken source-integrity or confidentiality checks.
- Do not alter the CDISC reference artifacts to make a test pass.
- Keep repository-authored code portable to SAS OnDemand where practical.
- Run repository validation before committing.
- For SAS changes, run the affected program and check in SAS Studio when access
  is available, then place redacted evidence under `outputs/`.
